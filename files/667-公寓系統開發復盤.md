## 一、步驟回顧（做了什麼 × 為什麼）

### 1. 需求釐清（帳單計算階段）
- 在對話中累積帳單，定義拆分規則：5/1 分界、天數比例（起訖皆計入）、÷3、四捨五入、尾差由管理者吸收。
- 意義：這是**業務邏輯定義**。先有正確的手算範例，程式才有對答案的基準。

### 2. 方案決策
- 三選一：A 單機 HTML／B 雲端共享／C 登入自助 → 選 C + 管理者更新繳費 + Bluehost PHP/MySQL。
- 意義：「各看各的」必然需要登入與資料庫；「誰標記已繳」決定權限設計。先決策再寫 code，避免重寫。

### 3. 資料庫設計（schema.sql）
- 四張表：`users`（帳號）／`settlements`（結算期）／`bills`（帳單）／`payments`（期 × 房客的繳費狀態）。
- 意義：每張表對應一個真實概念（正規化）。日後加人、加期數不需改結構。

### 4. 後端開發（PHP）
- 登入驗證、權限隔離（房客只查自己）、CSRF 防護、分攤計算函式。
- 關鍵技術點：
  - **bcrypt 雜湊**：密碼為不可逆亂碼，DB 看不到原碼 → 改密碼需先算雜湊再寫入。
  - **PDO prepared statements**：SQL 與資料分離，防注入。
  - **計算邏輯只寫一份**（functions.php）：房客頁、admin 頁、即時試算共用，改規則只改一處。

### 5. 部署（cPanel 四件套）
1. 建資料庫（庫＝倉庫）
2. 建 DB 使用者 + ALL PRIVILEGES（使用者＝鑰匙，授權＝配鑰匙）
3. phpMyAdmin 匯入 schema.sql
4. 上傳檔案 + 填 config.php
- 少任何一步 → HTTP 500。

### 6. 踩坑與修復（最有價值）
| 事件 | 根因 | 學到 |
|---|---|---|
| 按登入才 500 | DB 連線失敗 | 登入頁正常但提交壞＝連線問題，看 error_log 定位 |
| config.php 被當純文字輸出 | 檔案少了 `<?php` 開頭 | PHP 不執行就會吐原始碼；**密碼外洩的標準處置是立刻更換**，不是遮掩 |
| .htaccess 的必要性 | 沒上傳隱藏檔 | 它正是擋外部直接讀設定檔的防線（Mac 顯示隱藏檔：Cmd+Shift+.） |

### 7. 迭代優化（上線後）
- 手機 2×2 費用卡片（大數字 + 小標題 + 期間）、期間選擇器
- 西元年顯示（改一個 roc() 函式全站生效）
- 角色更名 Aries（程式字串 + DB 既有資料兩邊都要改）
- 圓形 logo、已繳 Thank You 貼圖（去背處理後僅已繳時顯示）
- 模式：每次只改最小範圍檔案、覆蓋上傳 → 最簡形式的持續交付。

## 二、實際接觸的技術棧

| 層 | 技術 | 實際操作 |
|---|---|---|
| 前端 | HTML / CSS / 原生 JS | 卡片版面、即時試算、複製按鈕 |

## 三、可複用場景

系統本質：**登入 → 各自看自己的資料 → 管理者輸入 → 自動計算 → 狀態追蹤**

### 客戶工作場景（商業價值最高）
- **客戶請款／對帳入口**：客戶登入看自己的報價、應付款、付款狀態（房客→客戶、帳單→請款單，約八成程式碼沿用；可接 dghm-quote 流程下游）
- **專案進度查詢頁**：客戶登入看自己專案的里程碑與交付狀態
- **UAT 驗收追蹤**：客戶勾選驗收項，管理端看全貌

### 其他情境
- 社區管理費、家族共同支出、合租分攤
- 小型課程繳費（學生看學費、老師標記已繳）
- 任何「1 管理者 + N 個只看自己資料的使用者」場合

### 沿用時只需改三處
1. schema 欄位（資料概念）
2. compute_split() 計算規則（業務邏輯）
3. 頁面文案

登入、權限、CSRF、部署流程**全部不變**。
